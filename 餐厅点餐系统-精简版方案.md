# 小型餐馆点餐系统 - 精简版技术方案

> 生成时间：2026-02-25  
> 版本：v2.0 (精简版)  
> 适用场景：小型餐馆（约10个固定卡座 + 临时座位）

---

## 一、需求分析（精简版）

### 1.1 业务场景

小型餐馆特点：
- **规模小**：固定卡座约10个，高峰期路边加临时桌
- **人手少**：可能只有老板+1-2个服务员
- **成本低**：希望系统轻量、易维护、低成本
- **效率高**：点餐要快，后厨要能实时看到订单

### 1.2 用户角色

| 角色 | 使用设备 | 核心需求 |
|------|----------|----------|
| **顾客** | 手机扫码 | 自助点餐、不用排队等服务员 |
| **服务员** | Pad/手机 | 代客点餐、查看各桌状态、上菜确认 |
| **老板/管理员** | PC/手机 | 看营业额、菜品销量、管理菜单 |

## 二、技术选型（单体架构版）

### 2.1 为什么不用微服务？

| 对比项 | 微服务架构 | 单体模块化架构 |
|--------|-----------|--------------|
| 部署复杂度 | 高（多个服务） | 低（一个Jar包） |
| 运维成本 | 高 | 低 |
| 服务器要求 | 至少2台 | 1台即可 |
| 开发周期 | 长 | 短 |
| 适合规模 | 大型连锁 | 小型单店 |

**结论**：小型餐馆用单体架构完全够用，后期若开分店再考虑拆分。

### 2.2 推荐架构

```

├─────────────┴─────────────┴─────────────┴───────────────────┤
```

## 三、支付方案（简化版）

### 3.1 当前版本：收款码模式

```
简化支付流程：
┌─────────┐    ┌─────────┐    ┌─────────┐
│  顾客   │    │  系统   │    │  服务员  │
└────┬────┘    └────┬────┘    └────┬────┘
     │              │              │
     │   下单完成    │              │
     │─────────────>│              │
     │              │   生成收款码   │
     │              │─────────────>│
     │   展示收款码  │              │
     │<─────────────│              │
     │              │              │
     │   扫码支付    │              │
     │──────────────┼─────────────>│
     │              │              │
     │   确认到账    │              │
     │──────────────┼─────────────>│
     │              │   确认收款    │
     │              │<─────────────│
     │   支付成功    │              │
     │<─────────────│              │
```

**实现方式：**
1. 老板个人的微信/支付宝收款码上传到系统
2. 顾客下单后展示收款码图片
3. 顾客扫码付款
4. 服务员确认到账后点击"确认收款"
5. 订单状态变为"已支付"

**优点：** 零成本、零对接、即时到账  
**缺点：** 需要人工确认

### 3.2 后续版本：个人支付接口（可选）

**个人能否开通支付接口？**
- 微信支付：个人无法申请商户号，需要个体工商户或公司
- 支付宝：个人无法申请当面付接口

**低成本方案：**
1. **注册个体工商户**（免费或几十元）
2. **申请微信/支付宝商户号**（费率约0.6%）
3. **对接聚合支付**（如收钱吧、付呗，费率约0.38%）

**建议**：先用收款码模式跑通业务，月营业额稳定后再考虑对接支付接口。

```
│ • 本月营业额    │ • 滞销菜品     │ • 翻台率          │
│ • 时段分析      │ • 菜品销售趋势 │ • 人均消费        │

└────────────────┴────────────────┴───────────────────┘
```

### 4.2 核心统计SQL

```sql
-- 今日营业额
SELECT 
    DATE(created_at) as date,
    COUNT(*) as order_count,
    SUM(total_amount) as total_amount,
    SUM(pay_amount) as real_amount
FROM orders 
WHERE DATE(created_at) = CURDATE()
AND status = 4;  -- 已完成订单

-- 热销菜品TOP10
SELECT 
    d.name as dish_name,
    d.category_id,
    SUM(oi.quantity) as total_quantity,
    SUM(oi.subtotal) as total_amount
FROM order_item oi
JOIN dish d ON oi.dish_id = d.id
JOIN orders o ON oi.order_id = o.id
WHERE o.status = 4
  AND DATE(o.created_at) = CURDATE()
GROUP BY oi.dish_id, d.name, d.category_id
ORDER BY total_quantity DESC
LIMIT 10;

-- 各桌消费统计
SELECT 
    t.table_no,
    t.name,
    COUNT(o.id) as order_count,
    SUM(o.total_amount) as total_amount,
    AVG(o.total_amount) as avg_amount
FROM restaurant_table t
LEFT JOIN orders o ON t.id = o.table_id 
    AND o.status = 4 
    AND DATE(o.created_at) = CURDATE()
WHERE t.is_deleted = 0
GROUP BY t.id, t.table_no, t.name
ORDER BY total_amount DESC;

-- 时段销售分析（按小时）
SELECT 
    HOUR(created_at) as hour,
    COUNT(*) as order_count,
    SUM(total_amount) as amount
FROM orders
WHERE DATE(created_at) = CURDATE()
  AND status = 4
GROUP BY HOUR(created_at)
ORDER BY hour;
```

## 七、数据库设计（精简版）

### 7.1 核心表

```sql
-- 用户表
CREATE TABLE `sys_user` (
  `id` BIGINT PRIMARY KEY AUTO_INCREMENT,
  `username` VARCHAR(50) NOT NULL COMMENT '用户名',
  `password` VARCHAR(100) NOT NULL COMMENT '密码',
  `real_name` VARCHAR(50) COMMENT '真实姓名',
  `phone` VARCHAR(20) COMMENT '手机号',
  `role` TINYINT DEFAULT 1 COMMENT '角色：1服务员 2管理员',
  `status` TINYINT DEFAULT 1 COMMENT '状态：0禁用 1启用',
```

## 九、总结

### 9.1 方案亮点

1. **轻量架构**：单体应用，部署简单，一台服务器即可
2. **灵活桌台**：固定卡座 + 临时座位，适应小餐馆弹性需求
3. **简化支付**：收款码模式，零成本启动
4. **实时通知**：WebSocket 推送给后厨，不用人工传票
5. **数据统计**：营业额、热销菜、桌台分析一目了然

### 9.2 后续可扩展

- **支付升级**：后续可对接微信/支付宝商户号
- **会员系统**：增加储值、积分、优惠券
- **多门店**：数据分离，支持连锁经营
- **小程序**：打包成微信小程序，顾客使用更便捷
